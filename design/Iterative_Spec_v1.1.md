# GlossaHub 迭代功能规划说明书 (Version 1.1)

本规划书针对 GlossaHub 词条协同与版本管理平台 v1.1 迭代而制定，核心目标是实现 **版本变更对比中的“一键合并与同步 (Version Merge & Sync)”**，打通对比到同步的数据闭环。

---

## 1. 业务需求背景

在当前 v1.0 版本中，“词条变更对比”看板（ComparisonTab）已实现了源表 A 与基准表 B 之间的三态差异（新增 ADD、删除 DEL、修改 MOD）的精准对比。但对比后，用户面临着“看得见却改不着”的痛点：
*   **手动追溯低效**：若需同步版本差异，用户必须截屏或导出对比 CSV，再到“词条管理”中搜索对应的关键字（KW）逐个手动修改，效率极低。
*   **版本追赶困难**：当一个大版本（如 `CC750`）需要合入主干（如 `C706 2.1`）新开发的上千条翻译时，手动同步几乎是不可能完成的任务。

---

## 2. 功能需求设计 (v1.1)

### 2.1 交互视图升级

#### A. 列表操作列 (Row Action)
在对比差异大表的每一行右侧，新增 **“同步操作”** 列：
1.  **对于【新增 (ADD)】行**（仅在 A 中存在）：
    *   提供“写入基准表 B”按钮（合并图标）。点击后直接将该条目（含其所在页面、KW、中文及全语种翻译）复制新建到基准表 B 中。
2.  **对于【删除 (DEL)】行**（仅在 B 中存在，A 中已被删）：
    *   提供“基准同步删除”按钮（垃圾桶图标）或“还原到源表 A”按钮。使用户既可选择让 B 对齐 A 将其移除，也可一键拉回误删的词条。
3.  **对于【修改 (MOD)】行**（A 和 B 翻译不同）：
    *   提供“覆盖到 B”或“回滚到 A”双向同步按钮，高亮展示 values 的 old ➔ new 覆盖。

#### B. 顶部工具栏“批量合并” (Batch Merge Control)
在表格上方增加复选框与批量合并控制栏：
*   **多选勾选**：允许用户通过左侧复选框，任意勾选若干差异项。
*   **一键合并按钮**：点击“一键同步所选差异到基准表”。
*   **安全二次确认**：弹窗提示确认框，如：`系统检测到您勾选了 12 条差异，是否确认一键将所选变动批量同步更新到目标表 [CC750]？`。

## 3. 技术方案设计

### 3.1 接口层设计
新增批量合并同步 API 路由：`POST /api/versions/sync-terms`。

#### A. Request Payload
```json
{
  "sourceVersionId": "version-a-uuid",
  "targetVersionId": "version-b-uuid",
  "syncActions": [
    {
      "type": "ADD",
      "kw": "KW_LAUNCHER_VISITOR",
      "data": {
        "context": "首页",
        "owner": "王赵云",
        "zh_cn": "游客",
        "translations": "{\"EN\":\"GUEST\",\"FR\":\"INVITÉ\"}"
      }
    },
    {
      "type": "MOD",
      "kw": "KW_LAUNCHER_ROAD_RIDING",
      "data": {
        "context": "首页",
        "zh_cn": "公路骑行",
        "translations": "{\"EN\":\"Road Cycling\",\"FR\":\"Vélo route\"}"
      }
    },
    {
      "type": "DEL",
      "kw": "KW_OLD_DELETED_KEY"
    }
  ]
}
```

#### B. SQL 事务执行序列
为了保障跨表写入的数据一致性，后端接收后必须包裹在标准 SQLite/PostgreSQL `TRANSACTION` 中：
```sql
BEGIN TRANSACTION;
-- 1. 遍历 syncActions
-- 1a. 对于 ADD/MOD 操作，使用 INSERT OR REPLACE 写入目标表
INSERT OR REPLACE INTO terms (id, version_id, kw, context, owner, zh_cn, translations, created_at, updated_at)
VALUES ($id, $targetVersionId, $kw, $context, $owner, $zh_cn, $translations, $now, $now);

-- 1b. 对于 DEL 操作，从目标表中物理移除
DELETE FROM terms WHERE version_id = $targetVersionId AND kw = $kw;

-- 2. 写入审计日志 logs_v2
INSERT INTO logs_v2 (timestamp, action, details, version_name, user_id)
VALUES ($now, 'SYNC_MERGE', '一键合并了版本间的变更项', $targetVersionName, $userId);

COMMIT;
```

### 3.2 审计日志对齐
每一次的一键同步或单行合并，系统均会在"变更记录（LogsTab）"中留存痕迹，其 `action` 标记为 `[同步合并]`，`details` 详细记录包含的同步 KW 列表，保障任何跨表操作都透明可审计。

## 4. 翻译记忆库（Translation Memory）

### 4.1 业务需求背景
在当前版本中，每个固件版本（如 `CC750`、`C706 2.1`）的词条翻译相互独立。当创建新版本时，翻译人员需要对相同 KW 重新翻译或逐条手动复制，重复劳动严重，交付周期被拉长。

### 4.2 功能需求设计

#### A. 新版本自动继承
创建新版本时，系统自动将基准版本的全部词条及翻译数据复制到新版本，翻译人员无需重复劳动，仅关注增量变更。

#### B. 跨版本翻译参考
在"智能翻译"表格中编辑某条词条时，右侧侧边栏展示其他版本中相同 KW 的翻译参考：
*   显示版本名称、对应语种译文、最后修改人与修改时间。
*   提供"应用此翻译"按钮，一键填充当前单元格。

#### C. 批量继承
在版本管理页面，新增"从版本 X 导入翻译"入口：
*   弹窗选择源版本与目标版本。
*   仅导入目标版本中**缺失翻译**的语种（不覆盖已翻译内容）。
*   导入完成后提示："共从 [C706 2.1] 继承了 N 条词条的翻译"。

### 4.3 技术方案设计
*   新版本创建时执行 `INSERT INTO terms ... SELECT ... WHERE version_id = $sourceId` 批量复制。
*   新增 `GET /api/versions/:versionId/terms/:kw/references` 接口，返回跨版本翻译参考数据。

## 5. 词条审核工作流

### 5.1 业务需求背景
当前版本的编辑模式为"改即生效"，AI 翻译结果也直接写入正式数据。翻译质量缺乏兜底机制，错误翻译可能直接进入交付物。

### 5.2 功能需求设计

#### A. 词条状态机
为每条词条引入 `status` 字段，状态流转如下：
```
待翻译 (DRAFT) → 翻译中 (TRANSLATING) → 待审核 (PENDING_REVIEW) → 已审核 (APPROVED) → 已发布 (PUBLISHED)
                                                                    ↓
                                                              驳回 (REJECTED)
```

#### B. 审核操作
*   **审核人角色**：可批量"通过"或"驳回"选中的词条。
*   **驳回原因**：驳回时必须填写原因（文本输入），被驳回的词条回到翻译人员的待办列表。
*   **状态过滤**：表格顶部的筛选栏增加"按状态筛选"下拉。

#### C. 视觉标识
表格中不同状态的行用左侧色条区分：
*   灰色：待翻译
*   蓝色：翻译中
*   橙色：待审核
*   绿色：已审核
*   红色：被驳回

### 5.3 技术方案设计
*   `terms` 表新增 `status VARCHAR(20) DEFAULT 'DRAFT'`、`reviewed_by VARCHAR(64)`、`reviewed_at TIMESTAMP`、`reject_reason TEXT` 字段。
*   新增 `PUT /api/terms/:termId/review` 接口，支持 `approve` / `reject` 操作。

## 6. 词条锁定机制

### 6.1 业务需求背景
已审核确认的词条，可能被其他翻译人员误改，导致已交付的翻译质量回退。需要一种"冻结"能力，保护已确认的数据。

### 6.2 功能需求设计

#### A. 行级锁定
*   在表格操作列增加锁形图标，点击即可锁定/解锁当前行。
*   锁定后，该行所有语种单元格变为只读，编辑入口隐藏。
*   锁定状态在表格上有视觉标识（行尾挂锁图标 + 浅灰底色）。

#### B. 权限控制
*   仅"负责人"角色可执行锁定/解锁操作。
*   普通翻译人员对锁定行仅可查看，如需修改需提交"解锁申请"。

### 6.3 技术方案设计
*   `terms` 表新增 `is_locked BOOLEAN DEFAULT 0`、`locked_by VARCHAR(64)`、`locked_at TIMESTAMP` 字段。
*   编辑接口（`PUT /api/terms/:termId`）校验 `is_locked`，锁定时返回 403。

## 7. 修改日志增强：操作可回退

### 7.1 业务需求背景
当前的"修改日志"是只读时间线，仅展示"谁改了什么"。翻译人员改错词条后无法快速回退，需要手动逐条改回，极易出错。

### 7.2 功能需求设计

#### A. 翻译前快照
每次词条修改，系统自动保存修改前的完整数据快照（KW、中文、全语种翻译）。

#### B. 一键回退
*   日志详情中增加"回退到此版本"按钮。
*   点击后弹窗二次确认："确认将词条 [KW_XXX] 回退至修改前的状态？"
*   回退操作本身也产生一条新日志（`action: 回退操作`），记录回退的源日志 ID。

#### C. 日志筛选
支持按操作人、时间范围、KW 关键字、操作类型进行组合筛选。

### 7.3 技术方案设计
*   新增 `term_snapshots` 表：`id, term_id, kw, zh_cn, translations, snapshot_at, source_log_id`。
*   每次 `PUT /api/terms/:termId` 写入前，先将旧值插入快照表。
*   新增 `POST /api/logs/:logId/rollback` 接口，从快照恢复数据。

## 8. 批量操作扩展

### 8.1 业务需求背景
当前仅支持"批量翻译"和"批量新增"。翻译人员在版本管理中存在两类高频痛点：
1.  某批词条的"所在页面"分类错误，需批量修正。
2.  需将当前版本的词条复制到其他版本，作为翻译起点。

### 8.2 功能需求设计

#### A. 批量设置字段
*   勾选若干词条后，点击工具栏"批量设置"。
*   支持批量修改：所在页面（`fld_page`）、字号类别（`fld_size`）等分类字段。
*   二次确认弹窗："确认将选中的 N 条词条的【所在页面】统一设置为【设置页】？"

#### B. 批量复制到其他版本
*   勾选若干词条后，点击"复制到其他版本"。
*   弹窗选择目标版本。
*   **KW 去重检查**：复制前系统自动比对目标版本中已存在的 KW：
    *   若存在重复 KW，弹窗提示："检测到目标版本中已存在以下 N 条相同 KW 的词条：[列表展示 KW + 中文]。是否跳过重复项仅复制新增词条，还是覆盖目标版本中的同 KW 词条？"
    *   提供两个选项：「跳过重复项」/「覆盖目标版本」。
*   操作完成后写入审计日志。

### 8.3 技术方案设计

#### 批量设置字段
```json
// POST /api/terms/batch-update
{
  "termIds": ["term-1", "term-2"],
  "fields": { "fld_page": "设置页" }
}
```

#### 批量复制到其他版本
```json
// POST /api/terms/batch-copy
{
  "sourceVersionId": "version-a",
  "targetVersionId": "version-b",
  "termIds": ["term-1", "term-2"],
  "duplicateStrategy": "skip" | "overwrite"
}
```
后端在 `TRANSACTION` 中执行：
*   查询目标版本中已存在的 KW 列表。
*   根据 `duplicateStrategy` 决定跳过或 `UPSERT`。
*   返回操作结果摘要：`{ copied: 10, skipped: 3, overwritten: 2 }`。

## 9. 体验打磨（低优先级）

以下优化优先级较低，可在核心功能稳定后逐步迭代。

### 9.1 暗色模式
*   跟随系统主题或手动切换。
*   复用现有 CSS 变量体系（`--bg-primary`、`--text-primary` 等），补充暗色变量值。

### 9.2 移动端适配
*   优先适配"查看"和"审核"场景（表格横向滚动 + 卡片视图切换）。
*   编辑场景仍以桌面端为主。

### 9.3 空状态引导
*   各模块在无数据时展示引导文案 + 操作按钮。
*   示例：翻译表格无数据时显示"当前版本暂无词条，去创建第一条"。

### 9.4 加载骨架屏
*   将全局 spinner 替换为骨架屏（Skeleton），减少感知等待时间。

### 9.5 Toast 通知统一
*   移除所有 `alert()` 弹窗，统一替换为 Toast 通知。
*   Toast 区分 success / warning / error 三种样式。

### 9.6 表格列宽可调
*   翻译表格表头支持拖拽调整列宽。
*   用户调整后的列宽持久化到 localStorage。

### 9.7 翻译进度微指示
*   翻译表格每行增加微进度条或文案提示（如"12/16 语种已翻译"）。
*   帮助翻译人员快速定位未完成的词条。
