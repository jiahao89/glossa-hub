## 1. 方案源起：如何确认 Dify + 多维表格插件的解决方案

### 💡 原始诉求与挑战
在硬件固件和多语言 App 开发中，有大量专有名词（如“踏频”、“阻力值”）需要被精准翻译成 16+ 种语言。传统的机器翻译（如 Google/DeepL）不认识专业词汇，且没有**翻译记忆库**。

### ⚙️ 方案确认过程
1.  **Dify 角色定义**：
    *   用户输入的词条首先在 Dify **本地知识库**中检索，若命中则直接返回标准译文。
    *   若未命中，则交给大语言模型（如 GPT-4/Claude）结合上下文语义进行翻译。
2.  **输入与输出契约**：
    *   **输入**：`{ "text": "踏频", "target_languages": ["en", "es"] }`
    *   **输出**：格式化 JSON：`{ "en": "Cadence", "es": "Cadencia" }`
3.  **多维表格对接**：最初设想以“飞书多维表格”作为数据库，利用飞书插件将翻译结果动态回填到表格中。

## 2. 架构质变：为什么要改为独立版本？

随着开发的深入，我们发现飞书多维表格作为协同翻译底座存在以下致命瓶颈：

### 🚫 放弃飞书的三大原因
1.  **网络与生态限制**：不创建企业级飞书应用，就无法安全、稳定地拿到高频读写权限。
2.  **性能与速率限制**：飞书开放平台对 API 的请求频次有严格限制（Rate Limit），在大表同步（千级词条）时极易触发限流并报错。
3.  **协同冲突无解**：多人同时在大表上修改时，飞书 API 无法有效做行级悲观锁/乐观锁控制，容易造成数据覆盖。

### 🚀 独立版架构选择
我们做出了重要的重构决定：**脱离飞书，构建全栈独立 Web 应用**。
*   **前端**：Vite + React (Heroui/Tailwind) 提供丝滑的大网格单页操作体验。
*   **后端**：Node.js Express 安全代理所有 API 请求。
*   **数据库**：支持双驱动，本地开发即开即用 SQLite (`glossahub.db`)，云端部署无缝切换到 PostgreSQL。

## 3. 交互奠基：菜单与功能结构的确认

为了能清晰管理复杂的多语言翻译状态，我们设计并确认了**单页 Tab 式工作区结构**：

```
📁 GlossaHub (工作区)
├── 📊 Dashboard (仪表盘看板)      --> 翻译/审核完成率、AI Token 消耗折线图
├── 🔤 Translation (词条大网格)    --> 增删改查主界面，支持双击编辑、AI 单点翻译
├── 🔍 Comparison (版本对比)       --> 双版本差异审计（ADD/DEL/MOD），过滤假差异
├── 📚 Glossary (专业词汇库)       --> csv 动态大表直接读写、导入与无损导出
├── 🗃️ Table Manager (数据表管理)  --> 数据表重命名、一键克隆与对齐
├── ⚙️ Languages (语种字典配置)    --> 可自定义 16+ 语种的显示顺序及头信息
└── 🕒 Audit Logs (日志与回退)     --> 记录每一次修改动作，支持 Diff 对比与回退
```

## 4. 数据底座：增加语种、数据表与专业词汇库

### 🌐 动态自适应语种与数据表
不同硬件项目（例如骑行码表 vs. 跑步机）的目标出口国家不同，翻译的语种也不同。
*   我们开发了**动态多列映射引擎**，所有大网格和 CSV 导入导出不绑定死任何语种。
*   支持通过 **Languages 页面** 动态增删语种。添加新语种后，数据库自动新增字段，主表动态增加列，无需修改前端代码。
*   支持数据表的重命名与克隆，方便快速基于旧版本拉出新的分支版本。

## 5. 安全协同：角色权限、审核机制与词条锁

### 🔑 权限防线 (RBAC)
*   **多项目隔离**：全路由挂载 `requireProjectMember` 拦截器，非项目关联成员返回 `403`。
*   **细粒度权限预置**：
    *   `Owner` (所有者) & `Editor` (普通用户)：可编辑翻译、发起 AI 批量翻译。
    *   `Viewer` (只读用户)：只能浏览和对比。
    *   `Admin` (系统管理员)：拥有系统设置、用户注册、解锁词条等最高特权。

### 🔒 词条锁与状态机
*   **锁定状态**：对于已通过最终测试、或者人工确认过的高精度译文，管理员可以对其进行“锁定” (Lock)。
*   **覆盖防护**：被锁定的词条在发起“批量同步”或“AI 覆盖翻译”时，会自动保持不变，防止高精度数据被机器翻译误覆盖。

## 6. 用户体验：UI 抛光与 AI 翻译标识

### 🤖 可视化追溯：AI 翻译标识
为了让翻译团队分清“哪些是机器翻译的，哪些是人工确认的”：
*   **Bot 图标指示**：凡是由 Dify/AI 翻译填充的单元格，左上角均显示一个精致的机器人 (Bot) 标识。
*   **无感布局**：图标被限制在 cell 的最左侧并置于顶层，人工双击修改编辑后，标识自动消失。

### 🎨 现代极客美学
*   **响应式暗黑模式**：基于 CSS 变量的一键黑白主题切换，降低长时间看多列大表时的眼部疲劳。
*   **微动画**：按钮加载、回退进度条、乐观锁保存时的浮动气泡，让协同过程富有动感。

## 7. 审计回滚：增加 Git 式版本回退与后悔药

### 🕒 全量日志审计
所有的增、删、改操作都会记录到 `logs` 表中，记录以下核心元数据：
`[操作时间] [操作人] [变动 KW] [中文原词] [操作类型: 修改/回退/同步] [变动详情 JSON]`

### 🔄 历史回退 (Rollback) & 后后悔药 (Undo)
为了防止协同过程中的误操作导致辛辛苦苦攒的翻译数据丢失：
1.  **回退面板**：在日志页面点击“回退”，可以调出该 KW 在过去所有时间点的快照。
2.  **后悔药机制**：当用户执行回退时，系统在写入旧值前，会**自动为当前值生成一份临时快照**。如果回退错了，还可以二次回滚，实现无损容灾。

## 8. 稳定性攻坚：三大核心 Bug 修复

在迭代到 `v1.1` 的过程中，我们重点修复了以下可能导致崩溃和数据丢失的严重 Bug：

| 故障现象 | 根本原因分析 | 修复方案 |
| :--- | :--- | :--- |
| **翻译同步会清空全表** | 批量导入或 AI 翻译同步时，若前端参数传入空数组，DELETE 条件失效触发了全表清空。 | 严格校验入参长度，若为 `0` 或 `null` 立即熔断并抛出异常，不允许执行 DELETE 语句。 |
| **云端 Postgres 连接崩溃** | PostgreSQL 中 implicit casting（隐式类型转换）非常严格，将 JSON 格式的字符串直接插到 `jsonb` 字段会抛出类型不匹配异常。 | 在 `server.cjs` 的 10 处涉及 `translations` 和配置字段写入的 SQL 中，全部追加 **`::jsonb`** 显式转换。 |
| **协同编辑相互覆盖** | A 和 B 同时打开同一行，A 修改保存后，B 不知情直接修改并保存，导致 A 的修改丢失。 | 引入**乐观锁**。前端保存时带上 `updated_at` 时间戳。若数据库中的最新时间戳不一致，则拦截写入并返回 `409 Conflict`。 |

## 9. 云端战役：多环境部署方案与网络故障讨论

### 🌐 云端三端部署拓扑
*   **前端**：托管在 **Vercel**（零配置快速构建）。
*   **后端**：托管在 **Render**（运行 Express Docker 容器，配置环境变量）。
*   **数据库**：托管在 **Supabase**（PostgreSQL 托管服务）。

### 📶 Render IPv6 网络不可达故障 (ENETUNREACH)
在将后端部署至 Render 后，连接 Supabase 数据库遇到了以下经典网络故障：
*   **报错**：`connect ENETUNREACH 2406:da12...:5432`
*   **原因**：Supabase 默认将域名解析到了 IPv6 地址，而 Render 的容器网络环境仅支持 IPv4，不支持 IPv6 寻址。
*   **解决**：将数据库直连端口修改为 Supabase 提供的专用 IPv4 Pooler 代理端口：
    *   **主机**：`aws-1-ap-northeast-2.pooler.supabase.com`
    *   **端口**：`6543`
