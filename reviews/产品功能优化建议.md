# GlossaHub 产品与功能优化建议

> 审查视角：产品经理 / 用户旅程 / 功能完整度

## 二、P0 · 核心流程缺失功能

### 2.1 词条审核工作流

> **用户故事**：作为翻译负责人，我希望翻译完成后需要审核才能生效，避免错误翻译直接进入正式版本。

**现状**：任何有权限的人编辑即生效，无审核/锁定机制。AI 翻译结果直接写入，存在质量风险。

**建议**：
- 词条状态机：`待翻译 → 翻译中 → 待审核 → 已审核 → 已发布`
- 审核人角色可批量"通过/驳回"
- 支持按状态过滤表格视图
- 被驳回的词条标记原因并回到翻译人员

### 2.2 翻译记忆库（Translation Memory）

> **用户故事**：作为翻译人员，我希望系统能记住我之前翻过的词条，新版本中相同的 KW 自动带出历史翻译。

**现状**：每个版本独立，相同 KW 在新版本中需重新翻译或手动复制。

**建议**：
- 创建新版本时，自动继承上一个版本的翻译数据
- 新增"从版本 X 导入翻译"功能
- 编辑词条时，显示其他版本中相同 KW 的翻译参考

### 2.3 词条锁定机制

> **用户故事**：作为翻译负责人，我希望已审核确认的词条被锁定，防止其他人误改。

**现状**：所有编辑即时生效，无锁定。

**建议**：
- 行级锁定/解锁（`is_locked` 字段）
- 锁定后编辑需解锁或提交"变更申请"
- 锁定状态在表格上有视觉标识

## 三、P1 · 高频痛点

### 3.1 搜索能力不足

> **用户故事**：作为翻译人员，我希望搜索时能区分"仅搜索未翻译"和"搜索全部"。

**现状**：单一文本搜索框，匹配 KW + 中文。无法按翻译状态、页面归属、修改时间等条件组合筛选。

**建议**：
- 搜索支持多字段：KW、中文、所在页面、字号类别、任意语种
- 高级筛选：按翻译状态（已翻译/未翻译/部分翻译）、按修改时间、按修改人
- 搜索结果高亮关键词

### 3.2 导出格式单一

> **用户故事**：作为交付人员，我需要导出不同格式的文件给不同团队——开发要 JSON、测试要 Excel、产品要 PDF。

**现状**：仅支持 CSV 导出（当前视图）。

**建议**：
- JSON（开发对接用，按 KW 为 key）
- Excel .xlsx（含格式、分 sheet 按语种）
- 选择性导出（勾选语种列、勾选版本）

### 3.3 修改日志缺乏可操作性

> **用户故事**：作为翻译人员，我改错了词条想撤销，但日志只展示"谁改了什么"，无法回退。

**现状**：日志是只读的时间线，无操作能力。

**建议**：
- 日志支持"回退到此版本"（保留翻译前快照）
- 支持按操作人、时间范围、KW 筛选
- 导出日志（Excel/PDF）供质量追溯

### 3.4 批量操作效率低

> **用户故事**：作为翻译人员，我有 50 条词条需要批量把中文改为统一格式，但现在只能一条条编辑。

**现状**：批量操作仅限于"批量翻译"和"批量新增"。

**建议**：
- 批量修改指定字段（如批量设置"所在页面"）
- 批量删除（勾选后一键删除）
- 批量复制到其他版本

## 四、P2 · 产品扩展能力

### 4.1 多项目管理

> **用户故事**：未来不止码表固件，还有 App、官网等多条产品线的翻译需求。

**现状**：硬编码 `proj-default` 单项目，Dashboard 和翻译表只展示默认项目。

**建议**：
- 项目切换器（顶部或侧栏）
- 每个项目独立的版本、语种、词汇表
- 跨项目搜索（全局词条检索）

### 4.2 翻译质量评分

> **用户故事**：作为翻译负责人，我想知道哪些翻译质量高、哪些需要复查。

**现状**：无质量评估机制。

**建议**：
- AI 翻译结果自动打分（置信度）
- 人工标记"存疑"词条
- Dashboard 展示翻译质量分布图

### 4.3 实时协作感知

> **用户故事**：两个人同时编辑同一个表格，我希望看到对方正在编辑哪一行，避免冲突。

**现状**：无实时协作感知，靠乐观锁防冲突但存在竞态。

**建议**：
- 显示"当前在线用户"头像
- 编辑某行时该行标记"XXX 正在编辑"
- WebSocket 推送实时变更

### 4.4 术语库联动

> **用户故事**：我建了行业术语库，希望翻译时如果词条命中术语库，自动提示标准译法。

**现状**：词汇表是独立模块，翻译时无联动。

**建议**：
- 编辑词条时，如果中文命中词汇表，侧边提示标准译法
- AI 翻译 prompt 中自动注入相关术语
- 词汇表变更时，提示受影响的已翻译词条

## 五、P3 · 体验打磨

| 需求 | 现状 | 建议 |
|---|---|---|
| **快捷键** | 无 | Ctrl+S 保存、Ctrl+F 聚焦搜索、Tab 切换单元格 |
| **暗色模式** | 仅一种主题 | 跟随系统/手动切换 |
| **移动端适配** | 仅桌面端 | 优先适配查看和审核场景 |
| **空状态引导** | 部分模块空表格无提示 | 每个空状态加引导文案 + 操作按钮 |
| **加载骨架屏** | 仅 spinner | 替换为骨架屏减少感知等待 |
| **Toast 通知** | 部分 alert() | 统一用 Toast，移除 alert 弹窗 |
| **表格列宽可调** | 固定宽度 | 支持拖拽调整列宽 |
| **翻译进度条** | Dashboard 有全局覆盖率 | 每行显示"16/16 语种已翻译"微进度条 |

## 六、商业化与增长

### 6.1 产品定位建议

当前产品已具备**行业垂直翻译管理平台**的雏形。建议明确两条增长路径：

| 路径 | 方向 | 价值 |
|---|---|---|
| **深度** | 做深骑行/运动硬件行业的翻译标准化 | 术语壁垒 → 行业标准 → 不可替代性 |
| **广度** | 将 GlossaHub 泛化为通用 SaaS 翻译管理工具 | 多租户 → 订阅制 → 规模化收入 |

### 6.2 关键指标体系（当前缺失）

| 指标 | 现状 | 建议 |
|---|---|---|
| 翻译效率 | 无统计 | 每人每日翻译条数 |
| AI 采纳率 | 无统计 | AI 翻译被直接采纳的比例 |
| 返工率 | 无统计 | 被修改/驳回的词条占比 |
| 交付周期 | 无统计 | 从新增版本到 100% 覆盖的天数 |

建议在 Dashboard 增加"翻译效能"区块，用数据驱动改进。

## 八、总结

GlossaHub 的核心翻译工作流已经跑通，AI 翻译集成是亮点。当前最大的产品风险是**翻译质量管控缺失**（无审核/锁定/记忆库）——翻译人员改完即生效，AI 翻译直接写入，缺乏质量兜底。

建议下一版本集中解决 P0 三项，让翻译从"能做"升级为"敢交付"：
1. **翻译记忆库**——让重复劳动归零
2. **审核工作流**——让质量有兜底
3. **锁定机制**——让已确认的不被误改
