## 一、致命级（需立即修复）

| # | 问题 | 位置 | 影响 |
|---|---|---|---|
| 1 | **PG 事务跨连接失效** | `server.cjs:1423-1448` | `BEGIN/COMMIT` 经由 `pgPool.query` 执行，每次 query 借不同连接，事务完全失效。PG 下批量导入部分失败会留脏数据 |
| 2 | **三套 Schema 严重分裂** | server.cjs / db_init_pg.sql / migrate_to_v2.cjs | 主键类型（TEXT vs VARCHAR vs UUID）、glossary 表缺 `headers`/`fields` 列、languages 表在迁移脚本中整表缺失。PG 部署必然崩溃 |
| 3 | **`.env` 未被 `.gitignore` 忽略** | `.gitignore` | README 指导用户创建 `.env` 存密钥，但 gitignore 没排除，`git add .` 会泄露密钥 |

### 1.1 PG 事务跨连接失效（#1）

`db.run('BEGIN TRANSACTION')` 经由 `pgPool.query` 执行。pg Pool 的语义是：每次 `query()` 借出一个 client、执行完立即归还。因此 `BEGIN`、中间 `DELETE/INSERT`、`COMMIT` 会落在**不同的 client/连接**上：

- `BEGIN` 在连接 A 开了一个悬挂事务并被归还入池
- 后续语句在 B、C… 各自自动提交
- `COMMIT` 在另一条连接上提交了一个空事务

结果：批量导入 glossary terms 的"先清空再批量插入"在 PG 下**既无原子性也无隔离性**，部分失败会留下空表或半量数据。SQLite 单连接下串行执行不会触发此 bug，属于隐藏的"双库不一致"致命缺陷。

**修复建议**：使用 `pgPool.connect()` 拿单一 client 再 `BEGIN/COMMIT`。

### 1.2 三套 Schema 分裂（#2）

| 表/字段 | server.cjs (SQLite) | db_init_pg.sql (PG) | migrate_to_v2.cjs (PG) |
|---------|--------------------|--------------------|------------------------|
| 所有主键 id | `TEXT` | `VARCHAR(64)` | `UUID` |
| `glossary_tables.headers` | 有 | **无** | 整表缺失 |
| `glossary_terms.fields` | 有 | **无** | 整表缺失 |
| 日志表名 | `logs_v2` | `logs` | `logs` |
| `versions.created_by` | 有 | 有 | **无** |
| `languages` 表 | 有 | 有 | **整表缺失** |

server.cjs:1338 `JSON.parse(t.headers)`、1427 行 `UPDATE ... SET headers` 在 PG 库上**必崩**。

**修复建议**：以 server.cjs 运行时逻辑为权威，统一 db_init_pg.sql 和 migrate_to_v2.cjs。

### 1.3 .env 未被 gitignore（#3）

`.gitignore` 全文没有 `.env` 相关条目。README 第 81-87 行指导用户创建 `.env` 写入 `JWT_SECRET` 等敏感值，一旦 `git add .` 密钥直接进版本库。

**修复建议**：补充 `.env`、`.env.*`、`!.env.example`。

## 二、高危级（安全与功能）

| # | 问题 | 位置 | 影响 |

| 4 | **越权访问/RBAC 完全未落地** | `server.cjs:380-395` | `authenticateToken` 只验 JWT，从不查 `project_members`。任意登录用户可删库、读写任意项目 |

### 2.1 越权访问（#4）

`authenticateToken` 中间件只校验 JWT 合法性并写入 `req.user`，**从未**查询 `project_members` 或比对 `req.user.role`。所有 `:projectId` 端点直接信任 URL 中的 `projectId`。

- `DELETE /api/logs`（L733）无角色校验
- `POST /api/sync-cleanup`（L659-662）空数组时 `DELETE FROM terms` / `DELETE FROM versions` 全表清空
- `role` 枚举（owner/editor/viewer）定义了却从未使用

**修复建议**：加 `requireProjectMember` 中间件，sync-cleanup 加 WHERE project_id 限定。

### 2.2 bitable 残留（#9）

`isDemoMode` 恒为 `false`（第 79 行初始化且无任何 setter 调用），但非 Demo 分支大量调用 `bitable.base.getTableById()` 等飞书 SDK API。`bitable` 从未导入或声明。

- `handleSyncAllTables`（L1656-1749）近 100 行死代码引用未定义的 `bitable`
- 用户点击"批量翻译"时，`getUntranslatedRecordsForTable` 进入 else 分支抛 ReferenceError，被 catch 吞掉后返回空列表，导致界面显示"该版本下没有未翻译词条"的**静默错误**

**修复建议**：移除所有 bitable 残留代码，确认批量功能走纯 API 路径。
